# API Gateway Placeholder

This directory is a placeholder for the future API Gateway. The gateway itself is **not** part of this codebase generation. It will be developed separately as part of the playground exercises.

Downstream services have been explicitly engineered to support gateway integration by delegating all cross-cutting infrastructure concerns to this layer.

---

## Gateway Core Responsibilities

The production-grade API Gateway developed here will implement:

1. **Reverse Proxy & Dynamic Routing**: Directing client traffic to the appropriate microservices based on paths (e.g. `/api/v1/auth/*` -> Auth Service, `/api/v1/products/*` -> Product Service).
2. **Security & Authentication**:
   - Validating incoming user JWT access tokens (issued by the Auth Service).
   - Rejecting requests with invalid, missing, or expired tokens.
   - Restricting routes using Role-Based Authorization (RBAC) (e.g., preventing customer roles from calling `POST /api/v1/products`).
   - Verifying external API Keys for machine-to-machine traffic.
   - SSL Termination.
3. **Observability & Request Metadata**:
   - Generating standard `X-Request-Id` headers for incoming client calls.
   - Propagating `X-Correlation-Id` headers for downstream distributed request tracing.
   - Request Logging (method, URL, status, response times, logs forwarded to Loki/Elasticsearch).
   - Metrics aggregation (interfacing with Prometheus/Grafana scrapers).
4. **Resilience**:
   - **Redis Rate Limiting**: Capping calls per client IP or user token to prevent DOS.
   - **Response Caching**: Caching static endpoint responses (e.g., category listings) in Redis.
   - **Circuit Breakers**: Tripping and failing fast if downstream microservices start throwing timeouts or 5xx errors.
   - **Retry Policies & Timeouts**: Retrying failed transient calls with exponential backoffs and enforcing request timeouts.
5. **Architectural Patterns**:
   - **Backend for Frontend (BFF)**: Providing tailormade aggregated JSON formats for Web/Mobile apps.
   - **API Versioning**: Handling route rewrites to target correct downstream endpoints.

## Gateway to Downstream Contract

Downstream microservices assume they are deployed behind a trusted API Gateway. Rather than duplicating JWT parsing or session checks, the microservices expect the Gateway to execute security validations and forward verified user identities using the following HTTP headers:

| Header Name | Type | Description |
|---|---|---|
| `X-User-Id` | String (UUID) | The verified unique identifier of the logged-in user. |
| `X-User-Role` | String (`admin` \| `customer`) | The verified role of the logged-in user. |
| `X-Request-Id` | String (UUID) | Unique ID generated by the gateway for tracking this individual client request. |
| `X-Correlation-Id` | String (UUID) | Request propagation ID for tracking requests across distributed systems. |
